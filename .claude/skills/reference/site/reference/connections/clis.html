    CLIs¶

    CLI tools are command-line programs that Claude can just run in the terminal. No MCP server, no config file, no context cost. Install the tool, and Claude figures out how to use it by reading --help or the documentation.

    CLIs are the most reliable connection method. They've been around for years, they're battle-tested, and Claude is remarkably good at discovering their capabilities on its own.

    Quick Answer

    Use this when: The tool you want to connect has a CLI available. This should be your first check.

    Avoid this when: The tool has no CLI, or you need persistent connections with managed auth (that's MCP territory).

    What CC changes: Claude runs CLI commands directly in bash. You don't need to teach it the tool. It reads --help, discovers what's possible, and figures out the right command.

    Default move: Always check for a CLI first. Install it, tell Claude what you want, and let it handle the rest.

    How CLIs work with Claude¶

    sequenceDiagram
    participant You
    participant Claude as Claude Code
    participant CLI as CLI Tool

    You->>Claude: "Convert this markdown to HTML"
    Claude->>CLI: pandoc --help (discovers usage)
    CLI-->>Claude: Usage documentation
    Claude->>CLI: pandoc CLAUDE.md -o output.html
    CLI-->>Claude: File created successfully
    Claude-->>You: Done. Here's what I generated.

    Note over Claude,CLI: Context cost: ZERO<br/>CLI runs in bash, not in context

    The key difference from MCPs: there's nothing to configure. No JSON file, no server process, no authentication flow. Claude runs the command in bash, gets the output, and keeps working.

    Installing CLI tools¶

    Claude can install tools for you. Just ask:

    Install pandoc

    Claude adapts to your operating system:

    OS Command
    macOS brew install pandoc
    Ubuntu/Debian apt install pandoc
    Windows choco install pandoc or winget install pandoc

    Once installed, the tool is available immediately. No restart needed.

    Claude using a CLI vs you using it directly¶

    Here's the difference. You don't need to know the command syntax. Claude does.

    You using pandoc directly: 

    pandoc --from=markdown --to=html --standalone \
  --metadata title="Feature Brief" \
  --css=style.css \
  -o feature-brief.html feature-brief.md

    You with Claude Code: 

    Convert feature-brief.md to a styled HTML page with a table of contents

    Claude runs pandoc --help, discovers the flags it needs, constructs the command, and executes it. If you asked for something fancier (PDF output, custom fonts, numbered sections), Claude would figure out those flags too.

    This is why CLIs are so powerful with Claude. Tools that used to require memorizing flags and reading man pages become natural language interfaces.

    CLI tools that matter for PMs¶

    Essentials¶

    Tool What it does PM use case
    gh (GitHub CLI) Full GitHub access from the terminal Create issues, review PRs, check CI status, manage releases
    pandoc Convert documents between any format Markdown to PDF, HTML, Word. Share documents in the format your stakeholders want.
    jq Process and filter JSON data Parse API responses, filter data exports, reshape structured data

    Development¶

    Tool What it does PM use case
    git Version control Already installed. Claude uses it constantly for branching, committing, diffing.
    npm / pip Package managers Install dependencies, run project scripts, manage toolchains
    docker Container management Run databases, test environments, preview builds

    Productivity¶

    Tool What it does PM use case
    jira (Jira CLI) Jira access from terminal Create tickets, update statuses, query backlogs
    linear (Linear CLI) Linear access from terminal Same as Jira CLI but for Linear users
    slack (Slack CLI) Slack operations Post messages, search channels, manage workflows
    ffmpeg Video/audio processing Extract clips from recordings, convert formats, create thumbnails

    A real workflow: GitHub CLI¶

    The GitHub CLI (gh) is the poster child for CLI-over-MCP. Here's why:

    Task: "Create a bug ticket for the dashboard loading issue"

    With gh, Claude runs: 

    gh issue create --title "Dashboard takes 12s to load on first visit" \
  --body "## Problem\nFirst-time visitors see a blank screen for 12 seconds...\n\n## Expected\nDashboard loads in under 3 seconds..." \
  --label "bug,performance" \
  --assignee "@me"

    Zero context cost. No MCP server running. No tool definitions loaded. Just a command, a result, done.

    Compare:

    gh CLI GitHub MCP
    Context cost Zero Low (tool definitions loaded)
    Setup brew install gh + gh auth login Add to .mcp.json, configure auth, restart
    Reliability Extremely high High (but depends on MCP server process)
    Capabilities Full GitHub API access Whatever the MCP author exposed

    For GitHub operations, the CLI wins on every dimension. This pattern holds for most tools that have mature CLIs.

    When CLIs fall short¶

    CLIs are the default choice, but they're not always the answer.

    CLI is wrong CLI is right
    The tool has no CLI (Figma, most analytics platforms) The tool has a well-documented CLI (gh, jira, pandoc)
    You need persistent auth that refreshes automatically Auth is a one-time login command
    You need Claude to proactively discover capabilities across many tools You know which tool you want Claude to use
    The integration requires complex multi-step API flows The operation maps cleanly to a single command

    The honest truth: CLIs cover maybe 30% of the tools PMs use day to day. But that 30% includes the highest-frequency operations (git, GitHub, document conversion, data processing). Start there.

    How Claude self-discovers CLI tools¶

    This is the part that surprises people. You don't need to teach Claude how to use a CLI tool. It teaches itself.

    flowchart TD
    A["You: 'Use pandoc to convert<br/>this to a PDF with a TOC'"] --> B["Claude runs:<br/><code>pandoc --help</code>"]
    B --> C["Claude reads the<br/>usage documentation"]
    C --> D["Claude constructs<br/>the right command"]
    D --> E["Claude runs:<br/><code>pandoc input.md -o output.pdf --toc</code>"]
    E --> F["Done. Zero teaching<br/>required from you."]

    style A fill:#1e293b,color:#f8fafc,stroke:#374151
    style B fill:#374151,color:#f8fafc,stroke:#4b5563
    style C fill:#374151,color:#f8fafc,stroke:#4b5563
    style D fill:#374151,color:#f8fafc,stroke:#4b5563
    style E fill:#e63f02,color:#fff,stroke:#b43200
    style F fill:#22c55e,color:#fff,stroke:#16a34a

    Claude reads --help output, man pages, and documentation. It understands flag syntax, subcommands, piping, and output formatting. For well-documented tools, Claude's usage is often more correct than what a human would type from memory.

    This self-discovery is a huge advantage over MCPs, where you're limited to whatever the MCP author chose to expose.

    Tips¶

    Install tools globally, not per-project. CLI tools should be available everywhere. Use your system package manager (brew, apt).

    Let Claude chain commands. Claude is good at piping output from one tool to another: 

    gh issue list --label "bug" --json title,labels | jq '.[] | .title'

    Check if a CLI exists before reaching for an MCP. Many services that have MCPs also have CLIs. The CLI is almost always the better choice when both exist.

    Don't pre-teach. Your instinct might be to explain the tool to Claude. Don't. Just say what you want done and let Claude figure out the tool. It's faster and often more accurate than your instructions would be.
