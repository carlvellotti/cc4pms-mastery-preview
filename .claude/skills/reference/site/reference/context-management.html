    Context Management¶

    The single most important skill in Claude Code. Everything else builds on this.

    Quick Answer

    Use this when: Every session. Context management is not a technique you switch on. It is the foundation under everything Claude Code does.

    Avoid this when: Never. Even "just a quick question" has a context cost.

    What CC changes: You stop thinking about prompts and start thinking about what Claude knows, when it knows it, and how much room it has left to think. This is the shift from prompt engineering to context engineering.

    Default move: Start every session fresh. Keep your CLAUDE.md lean. Monitor with /context. Compact or clear before quality degrades.

    The mental model shift¶

    You are not writing prompts. You are managing a scarce resource.

    Claude Code is not a chatbot. It is a context engineering tool with a smart agentic loop powering it. The gap between casual users and power users is not about writing better messages. It is about controlling what Claude knows, when it knows it, and how much of its attention budget you have left.

    PMs already practice this skill daily. You scope requirements, curate information for stakeholders, and decide what context a teammate needs before a meeting. The difference is applying that same discipline to an AI collaborator.

    "If you're still typing instructions into Claude Code like you're asking ChatGPT for help, you're missing the entire point." -- Diamant AI

    The context window is working memory, not storage¶

    The 200k token context window is not a hard drive. It is working memory. And it is smaller than you think.

    The real capacity:

    What eats it Typical cost
    System prompts and reserved allocation ~22.5% from the start
    MCP tool definitions 15k-70k+ tokens depending on how many you have enabled
    Your CLAUDE.md (loads every request) Varies, but loads on every single API call
    Conversation history Grows with every message

    Practitioners report roughly 120k usable tokens in practice. That is your whiteboard, and everything you put on it competes for the same attention.

    Think of it like a product's UX budget. You have limited real estate, and every element you add competes for attention. Overloading it degrades the entire experience.

    The smart zone and the dumb zone¶

    LLMs do not maintain consistent intelligence across their context window. Matt Pocock popularized this concept, and multiple practitioners have validated it: as the context fills up, Claude gets measurably worse.

    %%{init: {'theme': 'dark'}}%%
flowchart LR
    A["0-40%\nSmart Zone"] --> B["40-60%\nCaution Zone"]
    B --> C["60-80%\nDegradation"]
    C --> D["80%+\nDanger Zone"]

    style A fill:#16a34a,color:#fff,stroke:#15803d
    style B fill:#ca8a04,color:#fff,stroke:#a16207
    style C fill:#ea580c,color:#fff,stroke:#c2410c
    style D fill:#dc2626,color:#fff,stroke:#b91c1c
    Context usage What happens What to do
    0-40% (Smart Zone) Sharp reasoning, follows instructions, catches nuance Work normally
    40-60% (Caution Zone) Occasional lapses, may miss instructions Monitor closely, consider compacting
    60-80% (Degradation Zone) Repetitive responses, forgets earlier context, generic output Compact immediately or start fresh
    80%+ (Danger Zone) Hallucinations, contradicts itself, ignores instructions Stop. Clear. Start over.

    Why does this happen? HyperDev's analysis provides the explanation: that "free" context space is not wasted. It is where reasoning happens. The less thinking room Claude has, the lower the quality of its output. Claude Code's auto-compact trigger was moved from 90% to roughly 75% for exactly this reason.

    The counter-intuitive insight: stopping earlier actually extends productive session length.

    The three types of context poison¶

    When context degrades, it does not simply get "less good." It actively poisons future outputs.

    Poison type What it looks like Example
    Failed attempt residue Previous wrong approaches linger and bias future attempts You asked Claude to fix a bug three different ways. All three failed. Now Claude keeps circling back to those failed approaches instead of trying something new
    Stale context Outdated information in CLAUDE.md or docs contradicts the current state Your CLAUDE.md still says "we use REST" but you migrated to GraphQL two months ago. Claude keeps generating REST endpoints
    Noise accumulation Verbose tool outputs, long debugging logs, and conversational cruft dilute signal A test suite dumped 500 lines of output into context. Now Claude is referencing error messages from passing tests

    The core commands¶

    Three commands form the foundation. Use them constantly.

    /clear -- the nuclear reset¶

    Completely resets the conversation. Only your CLAUDE.md persists.

    Use it: between unrelated tasks, after a plan is accepted, when Claude produces confused output, after two failed corrections on the same issue.

    Do not use it: mid-task when you need to preserve conversation context.

    "Every time you're doing something new, clear. You don't need all that chat history in your tokens every time." -- Steve, Builder.io

    /compact -- the smart summarizer¶

    Summarizes conversation history to free tokens while preserving key information. You can customize it with specific instructions:

    /compact Focus on the API design decisions and the test results

    Use it: mid-task when context is getting heavy (50-60% usage), before switching phases within the same task.

    Do not use it: when switching to an unrelated task (use /clear instead), when Claude is producing confused output (compacting confusion does not fix confusion).

    Compaction loses nuance

    Multiple practitioners report that auto-compaction drops important details. "Auto compact has a habit of nuking nuance." Power users disable auto-compact and manage it manually. For most users, the default is fine, but if you are doing sensitive work, run /compact yourself with specific instructions about what to preserve.

    /context -- the x-ray¶

    Shows exactly what is consuming your token budget. Run it when you suspect bloat, and run it at the start of sessions to understand your baseline.

    The decision matrix¶

    Situation /clear /compact
    Switching to an unrelated task Yes No
    Mid-task, context getting heavy No Yes
    After accepting a plan Yes No
    Claude producing confused output Yes No
    Two failed corrections on the same issue Yes No

    The two failed corrections rule¶

    This heuristic comes directly from Anthropic's official best practices:

    If you have corrected Claude more than twice on the same issue in one session, the context is cluttered with failed approaches. Run /clear and start fresh with a more specific prompt that incorporates what you learned.

    "A clean session with a better prompt almost always outperforms a long session with accumulated corrections." -- Anthropic

    This is the AI equivalent of "fail fast." Do not sink more time into a broken session. Fire the contractor, get a new one, and give them better instructions from the start.

    The session lifecycle¶

    Every session has three phases. Managing the transitions between them is where most people lose quality.

    %%{init: {'theme': 'dark'}}%%
flowchart LR
    A["Phase 1\nClean Start"] --> B["Phase 2\nActive Monitoring"]
    B --> C["Phase 3\nStructured Handoff"]
    C -->|"new task"| A

    style A fill:#16a34a,color:#fff,stroke:#15803d
    style B fill:#ca8a04,color:#fff,stroke:#a16207
    style C fill:#6366f1,color:#fff,stroke:#4f46e5

    Phase 1: Clean start¶

    Start fresh rather than continuing bloated sessions. Selectively load only the context you need. A clean session with focused context always outperforms a stale one with "everything from yesterday."

    Phase 2: Active monitoring¶

    Monitor context usage passively through the status line. Run /context when you suspect bloat. Proactively compact at 50-60% usage. Do not wait for auto-compact.

    Phase 3: Structured handoff¶

    Before ending a session or switching contexts, capture state externally:

    • Run /compact with specific focus instructions
    • Have Claude write a summary to a progress file or handoff document
    • For long-running tasks, create a handoff document: what is done, what is next, key decisions made

    "AI context is like milk; it's best served fresh and condensed." -- YK

    The context loading hierarchy¶

    Not all context loads the same way or at the same time. Understanding this hierarchy is how you control what Claude knows and when.

    Mechanism When it loads Token cost Best for
    CLAUDE.md Every request Full content, always Universal project context (keep it lean)
    MCP tool definitions Session start Full definitions, always Use sparingly, disable unused MCPs
    Skills On invocation ~30-50 tokens at start; full on use On-demand workflows and knowledge
    Sub-agents On spawn Separate context window entirely Heavy-lifting tasks, verbose output
    @-referenced files On mention Full content, that request only One-time context loading
    Subdirectory CLAUDE.md When reading files in that folder Full content, on-demand Area-specific rules and context

    The pattern: move things down this list as far as possible. What must Claude know every session? That goes in CLAUDE.md. What does Claude need only for specific tasks? That goes in skills or reference files. What produces verbose output? That gets delegated to a sub-agent.

    What belongs where¶

    The fundamental question is not "what goes in my CLAUDE.md?" It is "what goes WHERE?"

    Bad Good
    500-line CLAUDE.md with everything you have ever told Claude 60-80 line CLAUDE.md with universal context, reference files for details
    Task-specific instructions in the root CLAUDE.md ("use RICE for prioritization") Task-specific instructions in a subdirectory CLAUDE.md or skill
    Full documents pasted inline Short summaries inline, full docs on demand via @reference or file reads
    Code conventions in a PM workspace root CLAUDE.md Code conventions in .claude/rules/ or subdirectory CLAUDE.md
    15 MCP servers enabled at all times Only the MCPs you actually use for this project enabled

    The filtering question¶

    For every piece of context, ask: "Is this relevant to EVERYTHING I do in this workspace?"

    • Yes: root CLAUDE.md
    • Yes, but it is a tool instruction, not project context: .claude/rules/
    • Only for specific types of work: subdirectory CLAUDE.md or skill
    • Only needed when I explicitly ask for it: reference file, loaded on demand

    The PM-specific challenge¶

    PM work is fundamentally different from developer work when it comes to context management. A developer's CLAUDE.md serves one type of work. A PM's workspace serves ten.

    Dev pattern (one type of work) PM pattern (ten types of work)
    "Use TypeScript strict" Role and team context
    "Run tests before commit" Product summary
    "Kebab-case filenames" Key preferences
    "ESLint rules" Pointers to detailed docs
    One set of rules covers everything Rules change by task type

    Monday morning you are doing competitive research. Monday afternoon you are writing a PRD. Tuesday you are analyzing data. Wednesday you are drafting a stakeholder update. Each of these tasks needs different context, and loading all of it all the time is a waste of whiteboard space.

    The PM CLAUDE.md playbook:

    1. Your role -- who you are, who you report to
    2. Product summary -- 2-3 sentences on what you are building and for whom
    3. Team overview -- one line per person
    4. Universal preferences -- tone, style, things Claude gets wrong without being told
    5. Key paths -- where things live in your workspace

    Everything else goes in subdirectory CLAUDE.md files, skills, or reference documents.

    CLAUDE.md vs .claude/rules/¶

    These two are often confused. The distinction is simple:

    CLAUDE.md = briefing about YOUR WORLD. "We target mid-market teams of 50-500." "Sarah prefers detailed specs." A human teammate needs to know this.

    .claude/rules/ = operating instructions for CLAUDE. "Use kebab-case for file names." "Delegate research to sub-agents." Only an AI tool needs to be told this.

    Rules files in .claude/rules/ can also be scoped to specific file patterns, so they only activate when Claude is working with matching files. This is powerful for project-specific conventions.

    Keeping it alive¶

    CLAUDE.md files go stale. Old instructions can confuse newer versions of Claude. Stale workarounds become liabilities.

    The maintenance loop:

    1. Start small (you just did this)
    2. When Claude repeats a mistake, add one line
    3. End of session: "What should change in my CLAUDE.md based on what we did today?"
    4. Prune regularly. If an instruction is not earning its space, cut it

    Anthropic's own teams use this exact pattern. Your CLAUDE.md evolves from actual usage, not guesswork.

    Advanced: progressive disclosure through skills¶

    When your CLAUDE.md grows past ~500 lines, the answer is not "try to trim harder." It is skills.

    Skills load on-demand. A skill description costs ~30-50 tokens at session start. The full instructions only load when the skill is invoked. A project with 20 skills and a lean CLAUDE.md will outperform one with a 2,000-line CLAUDE.md every time.

    Approach Context cost
    500 lines in CLAUDE.md 500 lines loaded every request
    Same content as 10 skills ~300-500 tokens of descriptions loaded, full content only on invocation

    The migration path is clear: anything in CLAUDE.md that is not universally relevant to every session becomes a skill.

    Advanced: sub-agents for context isolation¶

    The primary purpose of sub-agents is not parallelism. It is context isolation.

    When you spawn a sub-agent, that agent gets its own fresh 200k-token context window. It does the heavy lifting, sends back a summary, and your main session barely moves. This is covered in depth on the Sub-agents page.

    The decision: Do you need the journey (follow-up questions, iteration, building on it) or just the answer (summary, output, result)? If you just need the answer, delegate it.

    Common mistakes¶

    Mistake What to do instead
    Never checking /context Check it at session start and whenever context feels heavy
    Letting auto-compact handle everything Run manual /compact with specific instructions at 50-60%
    Continuing a session after two failed corrections /clear and start fresh with a better prompt
    Loading 15 MCP servers Disable unused MCPs. Use /context to see the cost
    Putting everything in root CLAUDE.md Use subdirectory CLAUDE.md files and skills for task-specific context
    Never pruning CLAUDE.md Run the maintenance loop. Cut what is not earning its space
    Saying "thanks" at the end of a session Every message reprocesses the entire context. Just close the session

    Plug & Play¶

    /humanizer Behavioral skill that changes how Claude writes. Example of on-demand context loading -- 400+ lines of rules that only load when invoked, not every session.

    /feature-brief Generate skill with progressive disclosure. Instructions in SKILL.md, template and examples in the folder. Full reference material loads only when the skill runs.
