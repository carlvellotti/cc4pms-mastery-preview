    Building Skills¶

    Skills are markdown files that give Claude Code expertise on demand. Instead of cramming everything into CLAUDE.md, you write focused skills that load only when needed. A skill is a YAML header plus instructions in a SKILL.md file inside .claude/skills/.

    Quick Answer

    Use this when: You find yourself rebuilding the same context or repeating the same instructions across sessions.

    Avoid this when: You only need something once. Just prompt it. Build the skill the second time.

    What CC changes: Skills turn tribal knowledge into reusable, shareable tools. What used to live in your head (or a Google Doc nobody reads) becomes something Claude can execute consistently.

    Default move: Start with a text-only skill. Add references, tools, and code only when the simple version isn't enough.

    The Anatomy of a Skill¶

    Every skill is a SKILL.md file inside a named folder in .claude/skills/:

    .claude/skills/
  feature-brief/
    SKILL.md        ← this is the skill

    The file has two parts: YAML frontmatter and markdown instructions.

    ---
name: feature-brief
description: "This skill should be used when the user wants to draft
  a feature brief from product notes or meeting transcripts."
---

## Instructions

Read the product notes in the current directory. Draft a feature brief
covering: problem statement, proposed solution, success metrics, risks,
and open questions.

Use bullet points for risks and open questions. Keep the brief under
2 pages.

    That's a working skill. You can run it right now with /feature-brief.

    Frontmatter Fields¶

    The YAML header controls how your skill behaves. Only description really matters. Everything else is optional.

    Field What It Does When You Need It
    name Display name (lowercase, hyphens, max 64 chars) Always include for clarity
    description Tells Claude what this skill does and when to use it Always. This is the most important field
    allowed-tools Restricts which tools Claude can use When the skill should only read, or only write, or never run code
    disable-model-invocation Forces manual /name invocation only Sensitive operations you don't want auto-triggered
    user-invocable Set to false to hide from the / menu Reference files that other skills read, not standalone commands
    argument-hint Shows hint text during autocomplete When the skill takes an argument like a file path or topic
    model Force a specific model When you need Opus for quality or Haiku for speed
    context Set to fork to run in an isolated subagent When the skill shouldn't see or pollute conversation history

    The Description Field Matters More Than You Think¶

    The description controls whether Claude auto-triggers your skill when a user says something relevant. A vague description means Claude won't know when to suggest it. A specific one means it works like magic.

    Bad Good
    "Process documents" "This skill should be used when the user wants to process meeting transcripts and extract action items, decisions, and follow-ups with owners and deadlines"
    "Help with research" "This skill should be used when the user wants to design a user research study, including assumption surfacing, interview guide design, and research plan compilation"
    "Writing tool" "This skill should be used when the user wants to rewrite AI-generated text to sound more human, removing filler words, cliches, and generic phrasing"

    Write descriptions in third person: "This skill should be used when..." Include the specific keywords a user would naturally say.

    CC Limitation

    Even with a perfect description, auto-triggering is unreliable (about 20% of the time by default). For important skills, use manual /skill-name invocation. If you need higher auto-trigger rates, look into forced evaluation hooks.

    File Structure Patterns¶

    A simple skill is just a SKILL.md. As skills get more complex, you add reference files that load on demand.

    Simple skill (most skills start here)¶

    feature-brief/
  SKILL.md

    Skill with references¶

    feature-brief/
  SKILL.md              ← orchestrates everything
  references/
    template.md         ← loads only when skill runs
    examples.md         ← good/bad examples for Claude to follow
    style-guide.md      ← voice and formatting rules

    Skill with code¶

    data-check/
  SKILL.md
  scripts/
    validate.py         ← Claude can run this
  references/
    thresholds.md       ← acceptable ranges, rules

    The key pattern: SKILL.md tells Claude what to do and where to find supporting files. Claude only loads those files when it needs them. This keeps your context window clean. A skill with 5 reference files doesn't cost anything until you actually run it.

    flowchart TD
    S["/feature-brief"] --> SK[SKILL.md loads]
    SK --> I[Read instructions]
    I --> N{Need reference?}
    N -->|Yes| R[Load references/template.md]
    N -->|No| W[Start working]
    R --> W

    CC Advantage

    Progressive loading is what makes skills better than putting everything in CLAUDE.md. Your CLAUDE.md loads every session regardless of what you're doing. Skills load only when invoked. A project with 20 skills and a lean CLAUDE.md will outperform one with a 2,000-line CLAUDE.md every time.

    Writing Good Instructions¶

    The markdown body of SKILL.md is where you tell Claude what to do. Write it like you're giving instructions to a smart colleague who's never done this specific task.

    Bad Good
    "Write a good feature brief" "Read the product notes in the current directory. Draft a feature brief with these sections: Problem Statement, Proposed Solution, Success Metrics, Risks, Open Questions. Use bullet points for risks. Keep it under 2 pages."
    "Analyze the data and make charts" "Read the CSV files in data/. For each metric, create a Jupyter notebook cell with a chart. Use line charts for trends, bar charts for comparisons. Include a markdown cell above each chart explaining what it shows."
    "Help me write better" "Read the draft the user provides. Flag: filler words (just, very, really, actually), passive voice, sentences over 25 words, cliches, and jargon. For each issue, show the original and a rewrite."

    The pattern: be specific about inputs (what to read), outputs (what to produce), and format (how to structure it).

    Variables for Dynamic Input¶

    Skills support variable substitution:

    ---
name: scope-feature
argument-hint: "<feature-name>"
---

The user wants to scope the feature: $ARGUMENTS

Read the codebase and produce a scoping brief covering:
- Affected files and modules
- New components needed
- Existing patterns to reuse
- Estimated complexity

    Now /scope-feature dark mode toggle passes "dark mode toggle" as the argument.

    Inject Live Data with Shell Commands¶

    Use !command syntax to pull in live context before Claude sees the skill. The commands go in the markdown body of your SKILL.md:

    ---
name: pr-summary
description: "This skill should be used when the user wants a summary of recent changes."
---

## Current State

!git status --short

## Recent Changes

!git log --oneline -10

## Instructions

Summarize the recent changes above into a brief status update...

    Claude sees the output of these commands as part of the skill. This is how you build skills that react to the current state of your project, like a PR summary skill that reads the actual diff.

    Restricting Tool Access¶

    By default, a skill can use any tool Claude has access to. Use allowed-tools to limit what it can do.

    ---
name: code-review
allowed-tools: Read, Grep, Glob
---

    This skill can read and search files but can't edit anything.

    Scenario Allowed Tools Why
    Review/audit Read, Grep, Glob Analyze without modifying
    Document generation Read, Write Create files, don't edit existing ones
    Research Read, WebSearch, WebFetch Gather info, don't touch the codebase
    Full workflow (don't set this field) Let Claude use whatever it needs

    CC Limitation

    Listing a tool in allowed-tools gives Claude access without asking for user permission. If your skill lists Bash, Claude can run shell commands without confirmation. Be intentional about what you grant.

    Activation: Why Skills Don't Always Trigger¶

    By default, Claude decides whether to suggest a skill based on the description matching what the user said. This works about 20% of the time. That's not a bug, it's just how LLM-based matching works with many skills competing for attention.

    Your options, from least to most reliable:

    Approach Reliability How It Works
    Auto-trigger (default) ~20% Claude matches description to user input
    Manual /skill-name 100% User types the command directly
    Forced evaluation hook ~84% A hook forces Claude to evaluate each skill YES/NO before responding

    Forced evaluation hooks use Claude Code's hooks system to intercept every prompt and check it against your skill descriptions. See the Anthropic hooks guide for setup.

    For most PM workflows, just use manual invocation. Type /feature-brief instead of hoping Claude suggests it. Save hook-based activation for teams with dozens of skills where discovery matters.

    The Graduation Path¶

    Not everything needs to be a skill. Here's when to move context from one level to the next:

    flowchart TD
    A["One-time prompt"] -->|"Doing it again?"| B["Skill"]
    B -->|"Need external data?"| C["Skill + MCP"]
    B -->|"Need isolation?"| D["Skill + Subagent"]
    A -->|"Always needed?"| E["CLAUDE.md"]
    E -->|"Over 500 lines?"| B

    The rule: If you're rebuilding the same context more than once, make it a skill. If your CLAUDE.md is over 500 lines, move reference sections into skills. Skills are cheaper on context and easier to maintain.

    Skill Scoping¶

    Skills can live at different levels depending on who needs them:

    Scope Location Who Sees It Use Case
    Personal ~/.claude/skills/ All your projects Cross-project workflows (your writing voice, daily standup)
    Project .claude/skills/ in repo Everyone who clones the repo Codebase-specific tools (PR template, deploy checklist)
    Plugin Installed via plugin system Where plugin is installed Distributed skill packs

    Project-level skills committed to git mean everyone on your team gets the same tools automatically when they clone the repo.

    Common Mistakes¶

    Mistake Why It Happens What to Do Instead
    Monolithic SKILL.md Trying to handle everything in one file Keep SKILL.md under 500 lines. Move reference material to references/
    Vague description "Helps with tasks" Be specific: what it does, when to use it, in third person
    One big skill instead of several focused ones Feels simpler to maintain Chunk it. /voice-linkedin and /voice-newsletter beat one generic /voice
    Everything in CLAUDE.md Doesn't know about skills yet Move on-demand knowledge to skills. CLAUDE.md is for always-on context only
    Over-broad tool access Didn't think about it Set allowed-tools to the minimum the skill needs
    Fighting auto-trigger It "should just work" It won't, reliably. Use manual /skill-name for important workflows

    Plug & Play¶

    Example skills from the course you can study and adapt:

    /feature-brief Simple generate skill. Takes product notes, outputs a structured brief. Good starting template.

    /mom-test-filter Review skill with specific criteria. Scans interview guides against three principles and rewrites violations.

    /meeting-aftermath Composition skill. Orchestrates 6 sub-skills in sequence (process transcript, reality-check, update people, update actions, draft follow-ups, status update).

    /humanizer Behavioral skill. Changes how Claude writes by loading specific rules about what to avoid and what to do instead.
