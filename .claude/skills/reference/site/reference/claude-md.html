    CLAUDE.md¶

    Your AI teammate has amnesia. Every session, it forgets everything: your role, the project, the constraints, how you like things done. CLAUDE.md fixes that. It's an onboarding document that loads automatically before you type a single word, so Claude shows up already knowing who you are and what you're working on.

    Quick answer

    Use this when: You want Claude Code to remember your project context, rules, and preferences across sessions

    Avoid this when: You need task-specific context for a single session (use @file or Skills instead)

    What CC changes: Without CLAUDE.md, you re-explain everything every session. With it, Claude shows up already briefed on your project, your standards, and your workflow

    Default move: Start with 10-20 lines (project name, stack, validation commands). Grow it from friction, not imagination

    Here's what a minimal one looks like:

    # Acme Dashboard

B2B analytics dashboard for mid-market SaaS companies.
React 18 + TypeScript frontend, Python FastAPI backend, PostgreSQL.

## Key directories
- `src/components/` — shared UI components
- `src/features/` — feature-specific modules
- `api/` — FastAPI routes and services
- `tests/` — pytest suite

## Validation
- Tests: `pytest tests/ -v`
- Lint: `npm run lint`
- Build: `npm run build`

## Rules
- Always create a git branch before making changes
- Run tests before committing
- Use existing component patterns — check src/components/ first

    That's 20 lines. It takes Claude from "what project is this?" to "I know what this is, how it's built, and how to verify my work." Every single session, automatically.

    This guide covers how to write a good one, what to put in it, what to leave out, and the PM-specific patterns that make it actually useful.

    The mental model¶

    CLAUDE.md is not a configuration file. It's not a prompt. It's onboarding documentation for an AI teammate that starts fresh every day.

    Think about it this way: every morning, a brilliant contractor shows up at your desk. They can do anything you ask. But they have no memory of yesterday. No memory of your codebase, your team, your product, your preferences, your users. You have to re-explain everything, every time.

    CLAUDE.md is the briefing packet you hand them when they walk in. The better the packet, the faster they're productive. A bad one (or none at all) means you spend the first 10 minutes of every session re-explaining context that should be automatic.

    For PMs, this is the lever. More powerful than any individual prompt you'll ever write. A good CLAUDE.md means Claude already knows your personas, your quality standards, your preferred document formats, and your product's architecture before you ask it anything.

    The core promise

    Write it once, Claude knows it forever. Every session. Automatically.

    The thinking room trade-off¶

    Here's the catch: CLAUDE.md loads into Claude's context window with every message. Every line you add is a line Claude has to process, and a line less available for actual reasoning.

    Hannah Stulberg nailed this: "More context loaded = less room to think = lower quality output."

    The numbers:

    • CLAUDE.md loads with every single API call, including the warmup when you start a session
    • If your CLAUDE.md eats 10-15% of the context window before you type anything, it's too big
    • Anthropic recommends keeping it under ~500 lines
    • Boris Cherny targets ~2,500 tokens
    • HumanLayer's research found a hard ceiling: frontier LLMs follow roughly 150-200 instructions with consistency. Claude Code's system prompt already uses ~50 of those. Your actual budget is about 100 instructions. Smaller models degrade exponentially; larger ones degrade linearly, but the ceiling exists either way
    • One user discovered a single word "thanks" at the end of a session cost 7% of their quota because the entire context (including CLAUDE.md) gets reprocessed

    There's another wrinkle. Claude Code injects a system message that literally says: "this context may or may not be relevant to your tasks." Claude is told to actively discount CLAUDE.md content that doesn't seem universally applicable. So task-specific guidance buried in CLAUDE.md gets deprioritized by design. Another reason to keep it general and move the specifics to Skills.

    You want Claude to know everything. But everything has a cost. That's the tension.

    The rule: Include what Claude needs for the current session's work. Move everything else to Skills (which load on-demand) or reference files (which load only when called with @).

    Think of it as the difference between what goes on your desk vs what goes in your filing cabinet. CLAUDE.md is the desk. Keep it clean.

    The memory hierarchy¶

    CLAUDE.md isn't one file. It's a stack of files that load in order, from broadest to most specific:

    flowchart TD

    B["👤 User / Global<br/><code>~/.claude/CLAUDE.md</code>"] --> C
    C["👤 User Rules<br/><code>~/.claude/rules/*.md</code>"] --> D
    D["📁 Project<br/><code>./CLAUDE.md</code>"] --> E

    More specific always wins. A rule in your project's .claude/rules/testing.md overrides a conflicting rule in your user-level ~/.claude/CLAUDE.md. A directory-level CLAUDE.md in src/api/ overrides the root CLAUDE.md for files in that directory.

    What PMs actually need to know¶

    Most PMs will use three of these:

    Level File What goes here
    User-level ~/.claude/CLAUDE.md Your personal preferences: communication style, formatting, role context. Travels with you across projects.
    Project-level ./CLAUDE.md The main one. Project context, tech stack, validation commands, team rules. Version controlled in git.
    Local override CLAUDE.local.md Your personal project preferences that shouldn't be in git. Things like "I'm the PM on this team" or API keys for your local dev environment. Gitignored by default.

    The enterprise level matters if your company deploys org-wide standards. Directory-level matters for large monorepos. But for most PM work, project-level CLAUDE.md is where 90% of the value lives.

    What goes in¶

    The three-section framework that works for any project:

    1. The "What": project context¶

    What this project is, how it's built, where things live. Claude needs this to navigate your codebase and make appropriate decisions.

    # TaskFlow

B2B SaaS for team task management. 3,200 paying customers,
primarily 10-50 person teams in tech and professional services.

## Tech stack
- Frontend: React 18 + TypeScript, Tailwind CSS
- Backend: Python FastAPI, SQLAlchemy ORM
- Database: PostgreSQL 15
- Infra: AWS (ECS, RDS, S3), deployed via Terraform

## Key directories
- `app/api/` — API routes (one file per resource)
- `app/models/` — SQLAlchemy models
- `app/services/` — business logic (never in routes)
- `web/src/features/` — React feature modules
- `web/src/components/` — shared UI components

    Keep this section factual and scannable. Bullet points and tables over paragraphs. Claude doesn't need a narrative about your company history. It needs to know what's in app/services/ and why.

    2. The "Domain": what makes your product unique¶

    The context that shapes how Claude thinks about your product. This is where PMs add the most value. Nobody else on the team is going to write this.

    ## Users
- **Team Leads** — power users, manage task workflows, care about visibility
- **Individual Contributors** — daily active, care about speed and simplicity
- **Admins** — configure integrations, manage permissions, care about control

## Product principles
- Speed over features — if it's slow, it doesn't matter what it does
- Progressive disclosure — simple by default, powerful when needed
- Team-first — every feature should make the team better, not just individuals

## Current focus
- Q1: Enterprise permissions overhaul (SOC2 compliance driver)
- Active experiment: simplified invite flow (31% → target 50% completion)

    This section is the PM's home turf. Nobody else on the team writes this stuff. And when Claude knows your personas, your principles, and your current focus, the output quality changes completely. Compare a PRD written with this context to one without. You'll never go back.

    3. The "Validation": how to verify work¶

    The single most impactful thing you can put in CLAUDE.md. Give Claude a way to check its own work, and quality jumps 2-3x.

    ## Validation
Before completing any task, verify:
- Tests pass: `pytest tests/ -v`
- Lint clean: `npm run lint`
- Build succeeds: `npm run build`
- If UI change: take a screenshot and describe what changed

    John Kim said it best: "If there's one tip that matters more than the other 49, it's this: give Claude a way to verify its own work."

    Even for non-code PM work, validation rules matter:

    ## Validation (PM workspace)
Before completing any document:
- Does it have measurable success criteria? (not just "improve X")
- Are assumptions explicitly stated?
- Is there a "what would change my mind" section?
- Run through anti-AI writing checklist

    PM-specific additions¶

    Beyond the three sections, PMs should consider:

    Content Why it matters Example
    Decision frameworks Claude applies YOUR prioritization logic "Use RICE scoring. Default weights: Reach 30%, Impact 30%, Confidence 20%, Effort 20%"
    Output format preferences Consistent document style "PRDs follow this structure: Problem → Users → Goals → Non-goals → Solution → Success Metrics → Risks"
    Quality standards Calibrates "done" "Success criteria must be measurable. 'Improve retention' is not acceptable. '85% 30-day retention for Growth plan' is."
    Team context Better stakeholder awareness "VP of Eng (Sarah) cares about technical debt. CEO (Marcus) wants everything quantified."
    Anti-patterns Prevent bad habits "Never present a single recommendation. Always model 3 options with tradeoffs."

    What stays out¶

    Everything that doesn't need to load every session should live somewhere else.

    Move to Skills (load on-demand)¶

    Skills are markdown files in .claude/skills/ that Claude reads only when triggered. They're the filing cabinet: detailed reference material that would bloat your CLAUDE.md but is invaluable when needed.

    CLAUDE.md says Skill handles
    "We do user research using the Mom Test framework" The full research methodology: question templates, analysis rubrics, synthesis framework
    "PRDs follow our standard format" The complete PRD template with examples, anti-patterns, and review checklist
    "Use RICE for prioritization" The full RICE scoring system with worked examples and edge cases

    The migration path is simple: when a section of your CLAUDE.md grows past 20-30 lines on a single topic, move it to a Skill. CLAUDE.md keeps the one-liner reference ("We use RICE"), the Skill has the details.

    Move to reference files (load with @)¶

    Any file Claude can read with @path/to/file.md doesn't need to be in CLAUDE.md. Use @ for:

    • Detailed specs or requirements docs
    • Meeting notes or research transcripts
    • Data files or analysis results
    • Anything that's relevant to one task but not every task

    The official include/exclude list¶

    Anthropic's own guidance on what belongs and what doesn't:

    Include Exclude
    Bash commands Claude can't guess Anything Claude can figure out by reading code
    Code style rules that differ from defaults Standard language conventions Claude already knows
    Testing instructions and preferred test runners Detailed API documentation (link to docs instead)
    Repository etiquette (branch naming, PR conventions) Information that changes frequently
    Architectural decisions specific to your project Long explanations or tutorials
    Developer environment quirks (required env vars) File-by-file descriptions of the codebase
    Common gotchas or non-obvious behaviors Self-evident practices like "write clean code"

    The pattern: include what Claude can't infer from the code itself. Exclude what it can.

    The bloat warning signs¶

    Your CLAUDE.md is too big if:

    • /context shows it consuming more than 10-15% before you type
    • You're scrolling to find things in it
    • It contains full templates instead of references to templates
    • There's narrative explaining why decisions were made (Claude needs the decision, not the history)
    • You have paragraphs where bullet points would work

    Shrivu Shankar put it well: "Keeping your CLAUDE.md as short as possible is a fantastic forcing function for simplifying your codebase and internal tooling."

    The 5-section structure for PMs¶

    If you're building a PM workspace (not just working in a codebase), this structure works. It's designed so Claude can orient from CLAUDE.md alone, without you explaining anything.

    # [Your Name] — [Role] at [Company]

## Who I am
Senior PM on the Growth team at TaskFlow. I own the self-serve
funnel (signup → activation → expansion) and the Growth plan tier.
Direct report to VP Product (Sarah Chen).

## What I own
- **Growth plan** — $1.2M ARR, 3,200 customers, 98% retention
- **Self-serve funnel** — signup, onboarding, invite flow, activation
- **Active experiment** — simplified invite flow (target: 31% → 50% completion)
- **Quarterly goal** — 15% expansion revenue increase by Q2 end

## How this workspace works
- `Projects/` — one folder per active project, each with a brief.md
- `Knowledge/` — company context, product docs, People/ profiles
- `Tasks/` — active.md (current), backlog.md (future)
- `Templates/` — reusable document templates
- `.claude/skills/` — PM skills (standup, meeting-prep, prd, status-update)

## Available skills
- `/standup` — morning briefing from GOALS.md + Tasks/ + project activity
- `/meeting-prep [person]` — pre-meeting brief from People/ profiles
- `/prd` — guided PRD creation (interviews you first, then generates)
- `/status-update` — weekly update from project context

## Current focus
- **This week:** Finalize invite flow experiment design, prep board deck
- **Blocked on:** Waiting for Data team to instrument invite_accepted event
- **Key risk:** Growth plan retention (98%) is based on n=9 — not enough signal

    The test: Start a fresh Claude session in this workspace and ask "what am I working on?" If Claude answers correctly from CLAUDE.md alone, you're done. If it can't, your CLAUDE.md is missing something.

    Modular organization¶

    As your CLAUDE.md grows, break it apart. You have two tools for this.

    @imports¶

    Pull content from other files into CLAUDE.md:

    # TaskFlow PM Workspace

@Knowledge/company.md
@Knowledge/product.md
@.claude/rules/writing-standards.md

## Current focus
...

    Claude reads the imported files as if they were part of CLAUDE.md. This keeps your root file clean: a table of contents, not a wall of text.

    .claude/rules/¶

    Modular rule files that auto-load alongside CLAUDE.md:

    .claude/rules/
├── writing-standards.md    # Anti-AI rules, voice guidelines
├── document-format.md      # PRD structure, one-pager format
├── product-principles.md   # Decision frameworks, quality standards
└── testing.md              # Validation requirements

    Each file can be scoped to specific file types using YAML frontmatter:

    ---
description: TypeScript component conventions
globs: ["*.tsx", "*.ts"]
alwaysApply: true
---

Use functional components with hooks. No class components.
Import shared components from @/components before creating new ones.

    Rules scoped with globs only load when Claude is working with matching files. This saves context for everything else.

    When to split vs keep together¶

    Keep in root CLAUDE.md:

    • Project overview (5-10 lines)
    • Directory structure
    • Validation commands
    • Current focus / active work

    Move to .claude/rules/:

    • Style guides and conventions
    • Document format requirements
    • Domain-specific rules (one file per domain)
    • Team-specific guidelines (PM rules vs engineering rules)

    Move to @imports:

    • Persona documents
    • Product specs
    • Company context that rarely changes

    Creating your first CLAUDE.md¶

    Pick whichever fits where you are right now.

    Method 1: Let Claude scan the project¶

    Already have a codebase? Run /init. Claude reads your project structure, package files, README, and existing docs, then generates a starting CLAUDE.md.

    It's a solid starting point. But it only captures what's in the code, not what's in your head. Always review and add the PM context (personas, principles, quality standards) that Claude can't infer.

    Method 2: Have Claude interview you¶

    Start a session and say:

    "Ask me 5 questions that would help you have a complete understanding of this project and how I work."

    Claude asks about your role, your goals, what "done" looks like, what mistakes to avoid. Your answers become the CLAUDE.md. This works especially well for new workspaces where there's no existing code to scan.

    Allie K. Miller's version is even broader: "Have Claude interview you, NOT about the project at hand, but who YOU are." Your role, your communication style, how you think about quality. This goes in your user-level ~/.claude/CLAUDE.md and travels with you across every project.

    Method 3: Start from scratch and grow¶

    Write the minimum viable CLAUDE.md: project name, tech stack, validation commands. Maybe 10 lines. Start working. When Claude does something wrong, add a rule. When you repeat yourself, capture it. When you explain context for the third time, write it down.

    Boris Cherny (core Claude Code team) does this: "My setup might be surprisingly vanilla. Claude Code works great out of the box, so I personally don't customize it much."

    The best CLAUDE.md files aren't designed. They're grown from real friction. Start small. Add when you feel pain.

    The quick-add shortcut¶

    Mid-conversation, type # followed by a rule:

    # When creating React components, always add a Storybook story

    Claude asks where to save it, updates your CLAUDE.md or rules, and you keep working. No context switching to edit files.

    Keeping it alive¶

    A CLAUDE.md is a living document. The moment you write it, it starts going stale.

    The self-correcting loop¶

    This is the most powerful maintenance pattern. When Claude does something wrong:

    1. Correct it in conversation
    2. Say: "Update the CLAUDE.md so you don't make that mistake again"
    3. Claude writes a rule for itself

    Boris Cherny calls this out specifically: "Claude is eerily good at writing rules for itself." Every correction becomes a permanent improvement. Over weeks, your CLAUDE.md evolves from a basic briefing into a comprehensive set of guardrails shaped by real work.

    Doc rot¶

    Matt Pocock coined this term for the slow decay of CLAUDE.md accuracy. Your project changes (new packages, new conventions, restructured directories) but CLAUDE.md still describes the old version.

    Outdated context is worse than no context. Claude confidently follows instructions that no longer apply, and the bugs compound.

    The fix: Build maintenance into your workflow.

    • After major changes: Update CLAUDE.md when you restructure the project, add dependencies, or change conventions
    • Weekly audit: Ask Claude "Are any of my CLAUDE.md files too long, redundant, or out of date?"
    • The 10-15% check: Run /context periodically. If CLAUDE.md consumes too much, prune it
    • Run /insights weekly: It analyzes your recent sessions and surfaces patterns: what's working, what's failing, where you're spending time. When it flags something useful, add a rule or trim one that isn't pulling its weight

    When Claude ignores your rules¶

    This happens. It's a known issue. The community describes it as working with "a genius who has dementia."

    Workarounds:

    1. Keep rules short and actionable. "Always run tests before committing" works. A three-paragraph explanation of your testing philosophy doesn't.
    2. Use Hooks for critical rules. Hooks run deterministic scripts that can't be ignored. If a rule absolutely must be followed (like "never push to main"), make it a Hook, not a CLAUDE.md rule. See Hooks.
    3. Reinforce in conversation. Type # remember: always run tests mid-session to bump a rule's priority.
    4. The two-failed-corrections rule. If you correct Claude twice on the same thing and it still doesn't stick, run /clear and start a fresh session. The context is polluted. Adding more instructions to a confused session makes it worse, not better.

    Advanced patterns¶

    Once you're comfortable with the basics, these are worth knowing about.

    Design context for visual work¶

    If you're building UIs or reviewing designs, add a design section to your project CLAUDE.md:

    ## Design context
- Style: clean, modern, generous whitespace
- Colors: primary #e63f02, secondary #f4d35e, dark #0f172a
- Components: use shadcn/ui, don't create custom components
- Reference: see screenshots in docs/design/ for visual baseline

    Four lines of design context. That's it. But Claude's visual output with this vs without it looks like two different tools.

    Scoped CLAUDE.md for workflows¶

    For complex, repeatable workflows (like a weekly status update), create a workflow folder with its own CLAUDE.md:

    Workflows/weekly-update/
├── CLAUDE.md           # Scoped instructions for this workflow
├── template.md         # Status update template
└── anti-ai-rules.md    # Writing voice guardrails

    The scoped CLAUDE.md tells Claude how to run this specific workflow. You cd into the folder, and Claude picks up the workflow context automatically.

    Hierarchical CLAUDE.md for large projects¶

    In a monorepo or large project, subdirectory CLAUDE.md files specialize the root one:

    • Root CLAUDE.md: Architecture overview, shared patterns, validation commands
    • api/CLAUDE.md: API conventions, endpoint patterns, auth requirements
    • web/CLAUDE.md: Frontend conventions, component patterns, accessibility rules

    Claude loads both the root and the relevant subdirectory file. The subdirectory one overrides conflicting rules from the root.

    Brandon Casci's take: "Think of them as executable specifications, not just guidelines."
