    Your PM OS¶

    A personal OS is the workspace where Claude Code knows your context, your work is organized, and your recurring processes get easier over time instead of staying manual. Without it, CC is just a chatbot. You ask something, it answers, and next time you start from zero. With it, every session builds on the last.

    Quick Answer

    Use this when: You want to move from ad-hoc Claude Code usage to a system that compounds over time.

    Avoid this when: You're looking for a step-by-step tutorial. This page is the philosophy. The course module walks you through building yours.

    What CC changes: CLAUDE.md is how CC knows your context. Skills turn recurring work into reusable commands. MCPs connect to your real tools. The folder structure keeps your work organized. Together, they form a system that gets smarter the more you use it.

    Default move: Start with context (CLAUDE.md + a few reference files). Add structure as your work demands it. Automate only what hurts to do manually.

    The Three Layers¶

    Every effective OS, regardless of how it's organized, has the same three layers. The specific folder structure doesn't matter. The layers do.

    flowchart TD
    subgraph OS["YOUR OS"]
        C["CONTEXT\nClaude knows who you are\n─────────────────\nCLAUDE.md, reference files,\nproject docs"]
        W["WORK\nWhere things get done\n─────────────────\nProject folders, drafts,\noutputs"]
        A["AUTOMATION\nWhat makes it compound\n─────────────────\nSkills, rules, MCPs,\nhooks, scripts"]
    end

    Context means CC doesn't start from zero. You don't re-explain your role, your company, or your preferences every session. CLAUDE.md and reference files carry that forward.

    Work means your outputs have a home. Projects live in folders. Drafts are findable. Nothing disappears into conversation history.

    Automation means recurring things get easier over time. The fifth time you do something takes less effort than the first. Skills, rules, MCPs, hooks, and scripts are the mechanisms.

    The Fractal Property¶

    The same three-layer pattern shows up at every scale. Not just your whole OS, but every area within it.

    Your Whole OS                       One Area (e.g., Content Creation)
┌─────────────────────┐             ┌─────────────────────────────────┐
│ Context             │             │ Context: voice files, brand     │
│ Work                │   same      │   guidelines, audience docs     │
│ Automation          │  pattern    │ Work: drafts, published pieces, │
└─────────────────────┘     →       │   content calendar              │
                                    │ Automation: dev centers,        │
                                    │   publishing workflows          │
                                    └─────────────────────────────────┘

    Zoom into any single area -- content creation, project management, research, data analysis, meetings -- and it has the same three layers at a smaller scale. Each area has its own context (the background info specific to that type of work), its own work artifacts (the outputs), and its own automation (skills and processes for that domain).

    Your OS is not one monolithic system. It's a collection of areas, each with its own version of the three layers, each at a different level of maturity. That's normal. That's how it's supposed to work.

    The Five-Level Progression¶

    The three layers describe what your OS is made of. The five levels describe how each part develops over time.

    flowchart LR
    L1["Level 1\nManual"] --> L2["Level 2\nDocumented"]
    L2 --> L3["Level 3\nSkilled"]
    L3 --> L4["Level 4\nConnected"]
    L4 --> L5["Level 5\nAutomated"]
    Level What It Means Example (Meeting Summaries)
    1. Manual You do it in a conversation every time "Summarize this meeting for me" with format instructions every time
    2. Documented Context is written down so CC does it better CLAUDE.md says "when summarizing meetings, pull out action items with owners and deadlines"
    3. Skilled It's a reusable command /process-transcript produces consistent output every time. See Skills reference.
    4. Connected The skill pulls from real tools via MCPs Skill reads calendar to know which meetings happened, checks task manager for existing items
    5. Automated It just runs Every evening at 5pm, summaries generated, actions filed, tomorrow's prep ready

    The key insight: Different parts of the same system sit at different levels. Your content creation might be level 5 (full pipeline with publishing workflows), while your personal finance is level 1 (you just installed the budget tool last week). That's fine. That's expected.

    Real Examples at Different Levels¶

    Person Area at Level 4-5 Area at Level 1-2
    Teresa Torres Daily planning (automated research digest, /today command pulls from Trello) File organization (structured folders, no automation for how files move)
    Arman Hezarkhani Document processing (skilled processing commands, connected to tools) Ongoing work management (system processes inputs, doesn't manage long-running work)
    Carl Vellotti Content creation (full pipeline, scheduled stats, publishing workflows, 70+ skills) Personal finance (just installed the budget MCP, barely started)

    Context Lives Where It's Useful¶

    Context doesn't all live in one place. It lives wherever it makes sense.

    Root level:      CLAUDE.md, reference files that apply to everything
                 (company info, preferences, global rules)

Area level:      Context shared across related work
                 (brand guidelines in a content area, style guide
                 in a writing folder, stakeholder notes in meetings)

Project level:   Context specific to one piece of work
                 (the client brief, the research, the requirements)

    CC can read any of it. You don't need to centralize context. Put it where it's most useful.

    Some people have a dedicated reference/ or llm-context/ folder. Others scatter context files wherever they're relevant. Both work. The question is whether CC can find and read the right context when it needs to, and your CLAUDE.md can index those files so CC knows where to look.

    What Makes Good Context¶

    Bad Good
    A resume or bio ("I'm a PM at Acme Corp with 8 years of experience") Opinions about how work should be done ("stakeholder updates should lead with decisions needed, not status")
    "I like Claude to be helpful" "Lead with the answer. Use bullet points. Always assign owners to action items."
    Everything in one massive CLAUDE.md Global rules in CLAUDE.md, area context in area folders, project context in project folders
    Static context written once Context that evolves. When you find yourself explaining the same thing twice, capture it.

    When NOT to Level Up¶

    Not everything should go to level 5. There is a real cost to moving up the progression. More to maintain. More to remember. More that can break. A level 5 automation that you built six months ago and forgot how it works is worse than a level 2 workflow you do manually but reliably.

    The question is not "how do I get everything to level 5." The question is "what's worth the investment?"

    flowchart TD
    Q["'Should I automate this?'"] --> F{How often?}
    F -->|Rarely| STOP["Stay at Level 1-2\nNot worth the investment"]
    F -->|Regularly| P{How painful?}
    P -->|Manageable| STAY["Stay current level\nFriction isn't high enough"]
    P -->|Genuinely painful| COST{Maintenance cost?}
    COST -->|Low| UP["Promote one level\nSkill or connection"]
    COST -->|High| WAIT["Wait until the pain\nclearly exceeds the cost"]

    The framework for deciding: only promote something when the pain of doing it manually genuinely exceeds the cost of building and maintaining the automation.

    Most things can live at level 1 or 2 forever and that's fine. A few level 4-5 workflows will handle most of your recurring work. The 80/20 rule applies hard here.

    The common failure mode: People try to automate everything on day one. They design the whole thing at once, get overwhelmed, and abandon it. The people who succeed build gradually. They try to do something, figure it out, and if they do it enough times, they formalize it. It's pull-based, not push-based.

    The System Grows From Use, Not From Planning¶

    This is the most important principle. You don't sit down and design your OS. You discover it through friction.

    Teresa Torres didn't create all her context files one afternoon. She built them gradually. Every time she found herself explaining something to CC that she'd need to explain again, she stopped and said "let's capture this in a context file." She'd tell CC to interview her about a topic and then write a file from what it learned. At the end of sessions, she asks "what did you learn about me that we should add to a context file?"

    That's the pattern:

    1. You do work in CC
    2. You notice friction (re-explaining context, doing something the same way every time, wishing CC knew something)
    3. You formalize what's worth keeping (context file, skill, rule)
    4. The next time is easier
    5. Repeat

    The system tells you what's worth formalizing. You don't have to guess upfront. The friction is the signal.

    The Universal Cycle¶

    Every area of work in an OS follows the same cycle:

    flowchart LR
    CTX["Context\n(was there)"] --> AUTO["Automation\n(did the work)"]
    AUTO --> OUT["Output\n(landed in project)"]
    OUT --> LEARN["Learn\n(what to formalize)"]
    LEARN --> CTX

    Context feeds the skill. The skill does the work. The output lives in the project. And you learn what's worth capturing for next time. This cycle applies to quarterly planning, meeting processing, data analysis, document creation, content production. The content changes. The cycle doesn't.

    Building Your OS: The Practical Path¶

    Week 1: Start with CLAUDE.md. Write your role, preferences, and current focus. Add one project folder with real files in it. Do real work.

    Week 2-3: Notice what you keep re-explaining. Capture it in context files. Notice what you keep doing the same way. Consider whether it's worth a skill.

    Month 2: You have 3-5 context files, a few project folders, maybe 1-2 skills. One area is at level 3. Most are at level 1-2. This is a working OS.

    Month 3+: Connect a tool via MCP if the manual input is genuinely painful. Build more skills as patterns emerge. Archive completed projects. Keep CLAUDE.md current.

    Bad Good
    Design the perfect folder structure on day one Start with CLAUDE.md and one project folder. Add structure when work demands it.
    Build 20 skills before doing any real work Do real work first. Build skills when you notice repetition.
    Try to reach level 5 everywhere Let most things sit at level 1-2. Promote only when the pain justifies it.
    Copy someone else's OS structure Your OS reflects your work. Use others for inspiration, not blueprints.

    Three Real Setups¶

    Different people, different structures, same principles.

    The Obsidian PM¶

    work/                          (Obsidian vault)
├── CLAUDE.md                  (global rules + context file index)
├── llm-context/               (focused reference files)
│   ├── business-profile.md
│   ├── target-audience.md
│   └── personal-profile.md
├── tasks/                     (own CLAUDE.md, runs /today)
│   ├── CLAUDE.md
│   └── tasks/, ideas/, inbox/
├── writing/                   (own CLAUDE.md, style guide)
│   ├── CLAUDE.md
│   └── article-series/
└── research/

    Context layer: llm-context/ with focused reference files, plus CLAUDE.md files in each subfolder. Work layer: tasks, writing, research. Automation layer: /today command that generates daily planning from Trello + academic research.

    The Processing Queue¶

    workspace/
├── CLAUDE.md                  (processing instructions)
├── inbox/                     (raw inputs land here)
├── processed/                 (handled items)
├── outputs/                   (finished artifacts)
│   ├── summaries/
│   ├── action-items/
│   └── drafts/
├── reference/                 (read-only context)
└── .claude/skills/            (processing skills)

    Context layer: reference/ folder (company info, team roles, style guide), never modified. Work layer: inbox-to-processed-to-outputs flow. Automation layer: processing skills for different input types.

    The Business Runner¶

    life-os/
├── CLAUDE.md                  (global rules, initiative index)
├── Work Journal/              (planning context)
│   ├── Business Context.md
│   └── SAO.md
├── Initiatives/               (per-initiative strategy)
├── Projects/                  (active deliverables)
├── Content Creation/          (full pipeline)
├── .claude/skills/            (70+ skills)
└── .mcp.json                  (8 tool connections)

    Context layer: Business Context, SAO, Initiative files scattered wherever they're relevant. Work layer: Projects, Content Creation, Work Journal. Automation layer: 70+ skills, 8 MCPs, daily/weekly rituals.

    The Maintenance Habits¶

    An OS that doesn't get maintained rots. But maintenance should be lightweight, not a chore.

    _temp/ folder: Pure scratch space. Throwaway outputs, files to copy-paste, quick looks. Delete the whole thing periodically.

    Archive completed projects. Don't delete old work. Move it to archive/. Still readable, not cluttering your active workspace.

    Keep CLAUDE.md current. Review it when your role changes, when you start a major new area of work, or when CC keeps making the same mistake. If it's working, don't touch it.

    End-of-session habit: Ask CC "based on what we just did, is there anything we should update in CLAUDE.md or any cleanup you'd suggest?" Takes ten seconds. It's how your OS learns from use.

    Common Mistakes¶

    Mistake Why It Happens What to Do Instead
    Over-design upfront "I need the perfect structure first" Start with CLAUDE.md and one project. The structure reveals itself through use.
    Automate everything on day one Excitement about the possibilities Do real work manually first. Automate when friction tells you to.
    One massive CLAUDE.md "Everything is important" Global context in CLAUDE.md. Area context in area folders. Project context in projects.
    Copy someone else's OS "Teresa's setup looks great" Her OS reflects her work. Yours needs to reflect yours. Use others for principles, not structure.
    Never maintain it "It's set up, I'm done" End-of-session check, periodic CLAUDE.md review, archive old projects
    Treat all areas equally "Everything should be level 4" 80/20. A few high-maturity workflows. Most things at level 1-2.

    Plug & Play¶

    /quarterly-goals Example skill: reads project context, asks three questions, writes a goals document. Demonstrates the full context-to-output cycle.

    /meeting-aftermath Composable skill: six sub-skills chained together. Demonstrates the orchestrator pattern for multi-step workflows.

    /sync-github Back up your OS to a private GitHub repo. Version history, backup, and cross-device access in one command.
