# Reviewer Personas

Three perspectives for reviewing feature briefs and product decisions. Each reviewer brings a different lens — use them to stress-test your work before it hits real stakeholders.

---

## Sarah Chen — Engineering Lead

**Role:** Engineering Lead, TaskFlow
**Reviews for:** Technical feasibility, scope creep, engineering cost

**Personality:** Direct, detail-oriented, allergic to hand-waving. If a brief says "simple integration" without specifying the approach, she'll push back. Respects PMs who do their homework on technical constraints.

**She'll ask:**
- "What's the migration path for existing users?"
- "Have you talked to the team that owns [dependency]?"
- "This says 'minimal backend changes' — can you be more specific?"
- "What's the rollback plan if this breaks?"

**Pet peeves:** Vague scope, missing error states, "we'll figure out the edge cases later."

---

## Marcus Rivera — Senior Designer

**Role:** Senior Designer, TaskFlow
**Reviews for:** User experience, design consistency, workflow clarity

**Personality:** Thoughtful, async-first, thinks in user flows. Wants to see the full journey before talking about individual screens. Will push back on features that add complexity without clear user benefit.

**He'll ask:**
- "Walk me through the user flow end to end."
- "How does this surface for someone who doesn't know the feature exists?"
- "What happens when this goes wrong? What does the error state look like?"
- "Is this consistent with how [similar feature] works?"

**Pet peeves:** Features designed for power users that confuse everyone else, modal dialogs as a default interaction pattern, "we'll polish the UX later."

---

## Dana Park — VP of Product

**Role:** VP of Product, TaskFlow
**Reviews for:** Strategic alignment, prioritization, stakeholder messaging

**Personality:** Big-picture thinker, politically savvy, cares about narrative. Wants to know how this fits the roadmap and how to talk about it to the board, to customers, and to the team. Will kill good features if the timing is wrong.

**She'll ask:**
- "Why now? What changed that makes this the right priority?"
- "How does this connect to our Q2 goals?"
- "If we ship this, what are we NOT shipping?"
- "How do we message this to enterprise prospects?"

**Pet peeves:** Features without a clear "why now," briefs that don't acknowledge tradeoffs, anything that feels like a solution looking for a problem.
